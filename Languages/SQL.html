1.1.1.0.1 COLUMNS
  • TABLE_CATALOG
    • The name of the catalog to which the table containing the column belongs. This value is always def.
  • TABLE_SCHEMA
    • The name of the schema (database) to which the table containing the column belongs.
  • TABLE_NAME
    • The name of the table containing the column.
  • COLUMN_NAME
    • The name of the column.
  • ORDINAL_POSITION
    • The position of the column within the table. ORDINAL_POSITION is necessary because you might want to say ORDER BY ORDINAL_POSITION. Unlike SHOW COLUMNS, SELECT from the COLUMNS table does not have automatic ordering.
  • COLUMN_DEFAULT
    • The default value for the column. This is NULL if the column has an explicit default of NULL, or if the column definition includes no DEFAULT clause.
  • IS_NULLABLE
    • The column nullability. The value is YES if NULL values can be stored in the column, NO if not.
  • DATA_TYPE
    • The column data type.
    • The DATA_TYPE value is the type name only with no other information. The COLUMN_TYPE value contains the type name and possibly other information such as the precision or length.
  • CHARACTER_MAXIMUM_LENGTH
    • For string columns, the maximum length in characters.
  • CHARACTER_OCTET_LENGTH
    • For string columns, the maximum length in bytes.
  • NUMERIC_PRECISION
    • For numeric columns, the numeric precision.
  • NUMERIC_SCALE
    • For numeric columns, the numeric scale.
  • DATETIME_PRECISION
    • For temporal columns, the fractional seconds precision.
  • CHARACTER_SET_NAME
    • For character string columns, the character set name.
  • COLLATION_NAME
    • For character string columns, the collation name.
  • COLUMN_TYPE
    • The column data type.
    • The DATA_TYPE value is the type name only with no other information. The COLUMN_TYPE value contains the type name and possibly other information such as the precision or length.
  • COLUMN_KEY
    • Whether the column is indexed:
      • If COLUMN_KEY is empty, the column either is not indexed or is indexed only as a secondary column in a multiple-column, nonunique index.
      • If COLUMN_KEY is PRI, the column is a PRIMARY KEY or is one of the columns in a multiple-column PRIMARY KEY.
      • If COLUMN_KEY is UNI, the column is the first column of a UNIQUE index. (A UNIQUE index permits multiple NULL values, but you can tell whether the column permits NULL by checking the Null column.)
      • If COLUMN_KEY is MUL, the column is the first column of a nonunique index in which multiple occurrences of a given value are permitted within the column.
      • If more than one of the COLUMN_KEY values applies to a given column of a table, COLUMN_KEY displays the one with the highest priority, in the order PRI, UNI, MUL.
      • A UNIQUE index may be displayed as PRI if it cannot contain NULL values and there is no PRIMARY KEY in the table. A UNIQUE index may display as MUL if several columns form a composite UNIQUE index; although the combination of the columns is unique, each column can still hold multiple occurrences of a given value.
  • EXTRA
    • Any additional information that is available about a given column. The value is nonempty in these cases:
1.1.1.0.2 KEY_COLUMN_USAGE

The KEY_COLUMN_USAGE table describes which key columns have constraints.

  • CONSTRAINT_CATALOG
    • The name of the catalog to which the constraint belongs. This value is always def.
  • CONSTRAINT_SCHEMA
    • The name of the schema (database) to which the constraint belongs.
  • CONSTRAINT_NAME
    • The name of the constraint.
  • TABLE_CATALOG
    • The name of the catalog to which the table containing the column belongs. This value is always def.
  • TABLE_SCHEMA
    • The name of the schema (database) to which the table containing the column belongs.
  • TABLE_NAME
    • The name of the table containing the column.
  • COLUMN_NAME
    • The name of the column that has the constraint.
    • If the constraint is a foreign key, then this is the column of the foreign key, not the column that the foreign key references.
  • ORDINAL_POSITION
    • The column’s position within the constraint, not the column’s position within the table. Column positions are numbered beginning with 1.
  • POSITION_IN_UNIQUE_CONSTRAINT
    • NULL for unique and primary-key constraints. For foreign-key constraints, this column is the ordinal position in key of the table that is being referenced.
  • REFERENCED_TABLE_SCHEMA
    • The name of the schema (database) referenced by the constraint.
  • REFERENCED_TABLE_NAME
    • The name of the table referenced by the constraint.
  • REFERENCED_COLUMN_NAME
    • The name of the column referenced by the constraint.

The REFERENTIAL_CONSTRAINTS table provides information about foreign keys.

  • CONSTRAINT_CATALOG
    • The name of the catalog to which the constraint belongs. This value is always def.
  • CONSTRAINT_SCHEMA
    • The name of the schema (database) to which the constraint belongs.
  • CONSTRAINT_NAME
    • The name of the constraint.
  • UNIQUE_CONSTRAINT_CATALOG
    • The name of the catalog containing the unique constraint that the constraint references. This value is always def.
  • UNIQUE_CONSTRAINT_SCHEMA
    • The name of the unique constraint that the constraint references.
  • UNIQUE_CONSTRAINT_NAME
    • The name of the column that has the constraint.

    • The value of the constraint MATCH attribute. The only valid value at this time is NONE.

  • TABLE_NAME
    • The name of the table containing the column.
  • REFERENCED_TABLE_NAME
    • The name of the table referenced by the constraint.

1.3.1.1 operational vs. analytical, & relational databases

Operational primarily used in online transaction processing (OLTP) key feature ability to collect modify data on daily basis stores

  • dynamic data and up to the minute info
  • transactional every day use, data change frequently

Analytical primarily used in online analytical processing (OLAP) key feature tracking trends, data over time, and ability to make predictions

  • stores static data (often pull from operational for data source)
  • another term is data warehouse, use for analysis not transactions, shows history over time, data does not change frequently, data often from operational database

a relationship is formed in the relational model implicitly through matching values of a shared field

SQL to create, maintain and query databases

  • You recieve data in a regional database by using Structured Query Language (SQL)
  • three components of basic SQL Query are the SELECT ... FROM statement, the WHERE clause, & the ORDER BY clause
  • SELECT indicates the fields selected, FROM is the table(s) within which the fields belong
  • the WHERE clause allows for filtering by criteria
  • the ORDER BY clause is for sorting in ascending or descending order
  • Queries can be saved
  • a simple scripting language that can be used to create, add, modify, and query relational databases
  • rough standard but kinda different dialects

RDMS or “Relational Database Management System”

  • software application program to create maintain modify and manipulate a relational database

  • A relational database stores data in relations, which the user perceives as tables. Each relation is composed of tuples, or records, and attributes, or fields. - The physical order of the records or fields is immaterial, each record in the table is identified by a field that contains a unique value

    • Built-in multilevel integrity (basically the issue checker)
    • logical and physical data independence from database applications (changes don’t auto break dependent things unlike excel refs)
    • guaranteed data consistency and accuracy
    • easy data retrieval
    • reduces non-strategic repetition of data (same info multiple sheets, multiple locations of info)
    • tables are connected to one another through strategic repetition called “keys” (matching info boxes)

1.3.1.2 Primary Keys & 1:1, 1:M, & N:M Relationships

a one-to-one relationship is when a single record in one table is related to zero or one record in the other table and vice-versa

a One-to-many relationship is where one record in one table is related to any number of records in the other table but the second table records only relate to one record in the first

you ALWAYS copy the primary key from the “1” side to the “many” side

without a table diagram (ERD) (entity-relationship diagram) use logic, ie. when a single record in the 1st table can be related to one or more records in the second table but a single record in the second table ONLY relates to one record in the first table

a many-to-many relationship is when a record in one table can be related to any number of records in the other table and vice-versa the relationship is established via a linking table (the two primary keys together form a composite primary key, but separately are foreign keys)

1.3.2.1 Elements of a Database

logical database describes the size shape and necessary systems for the database, then the physical database can be built

The relational database is based on two branches of mathematics known as set theory and first-order predicate logic.

the problems that good design avoids: redundant data, duplicate data, invalid data, absence of required data

  • The database supports both required and ad hoc information retrieval. (expected and unconventional usage)
  • The tables are constructed properly and efficiently. (single subject, unique values, )
  • Data integrity is imposed at the field, table, and relationship levels. (helps ensure validity)
  • The database supports business rules relevant to the organization.
  • The database lends itself to future growth.
1.3.2.1.1 Benefits of good design
  • The database structure is easy to modify and maintain. (modifications do not have spill effects)
  • The data is easy to modify. (few duplicates as well)
  • Information is easy to retrieve. (properly established relationships between tables.)
  • End-user applications are easy to develop and build.
    • data integrity

1.3.2.2 Database Design Steps/Phases

  1. Requirement analysis - examination of the business being modeled, needs, and requirements
  2. Data modeling using a method such as ‘entity-relationship (ER) diagramming, semantic-object modeling, object-role modeling, or UML modeling.’ as means of visually representing different aspects of the database structure such as tables, table relationships, and relationship characteristics
    1. each method incorporates a set of diagramming symbols to represent a databases structure and characteristics
    2. fields are also defined and associated with the appropriate tables during the data-modeling phase.
    3. each table is assigned a primary key various levels of data integrity are identified and implemented a relationships are established via foreign keys
  3. Normalization is the process of decomposing large tables into smaller ones in order to eliminate redundant and duplicate data and avoid problems with inserting, updating, or deleting data. during the process table structures are tested against normal forms (a specified set of rules to test for a particular set of problems)
    1. (The normal forms currently in use are First Normal Form, Second Normal Form, Third Normal Form, Fourth Normal Form, Fifth Normal Form, Sixth Normal Form, Boyce-Codd Normal Form, and Domain/Key Normal Form.)

A table type allows you to classify a table by the role it plays within the database and provides you with a means of identifying tables that function in a similar manner. The table’s role determines its type, and you can associate four table types with a given table:

  1. A data table represents a subject that is important to the organization and is the primary foundation of the information that the database provides.
  2. A linking table establishes a link between two tables in a many-to-many relationship.
  3. A subset table contains fields that are related to a particular data table and further describes the data table’s subject in a very specific manner.
  4. A validation table contains relatively static data and is a crucial component of data integrity.

a table description provides a clear definition of the subject represented by the table and states why the subject is important to the organization (there are guidelines that govern how to create these)

Make sure to resolve multivalued and multipart fields

Redundant data is a value that is repeated in a field as a result of the field’s participation in relating two tables or as a result of some field or table anomaly. In the first instance, the redundant data is appropriate; by definition, a field used to relate one table to another will contain redundant data. Redundant data is entirely unacceptable in the second instance, however, because it poses problems with data consistency and data integrity; therefore, you should always strive to keep redundant data to an absolute minimum.

A duplicate field is a field that appears in two or more tables for any of these reasons.

  • It is used to relate a set of tables together. (the only necessary case)
  • It indicates multiple occurrences of a particular type of value.
  • There is a perceived need for supplemental information.

1.3.5.1 To resolve multivalued fields

  1. Remove the field from the table and use it as the basis for a new table. If necessary, rename the field in accordance with the field name guidelines.
  2. Use a field (or set of fields) from the original table to relate the original table to the new table; try to select fields that represent the subject of the table as closely as possible. The field(s) you choose will appear in both tables.
  3. Assign an appropriate name, type, and description to the new table and add it to the Final Table List.

1.3.5.2 To refine previously unidentified subset tables

  1. Remove all the fields that the subset tables have in common and use them as the basis for a new data table.
  2. Identify what subject the new data table represents, and then give that table an appropriate name.
  3. Make sure that the subset tables represent subordinate subjects of the data table and modify the subset table names as necessary.
  4. Compose a suitable description for the data table and then add it to the Final Table List. Indicate the table type as “Data.”

1.3.5.3 Guidelines to creating table names

  • Create a unique, descriptive name that is meaningful to the entire organization.
  • Create a name that accurately, clearly, and unambiguously identifies the subject of the table.
  • Use the minimum number of words necessary to convey the subject of the table.
  • Do not use words that convey physical characteristics. (ie. file, record, table, etc.)
  • Do not use acronyms and abbreviations.
  • Do not use proper names or other words that will unduly restrict the data that can be entered into the table. (ie. location or other specification)
  • Do not use a name that implicitly or explicitly identifies more than one subject.
  • Do use the plural form of the name. (pluralize table names, singularize field names)

1.3.5.4 Guidelines to creating table descriptions

  • Include a statement that accurately defines the table.
  • Include a statement that explains why this table is important to the organization.
  • Compose a description that is clear and succinct.
  • Do not include implementation-specific information in your table description, such as how or where the table is used.
  • Do not make the table description for one table dependent upon the table description of another table.
  • Do not use examples in a table description.

1.3.5.5 Guidelines to creating field names

  • Create a unique, descriptive name that is meaningful to the entire organization.
  • Create a name that accurately, clearly, and unambiguously identifies the characteristic a field represents. (use prefixes where necessary)
  • Use the minimum number of words necessary to convey the meaning of the characteristic the field represents.
  • Do not use acronyms and use abbreviations judiciously.
  • Do not use words that could confuse the meaning of the field name.
  • Do not use names that implicitly or explicitly identify more than one characteristic.
  • Use the singular form of the name.

1.3.5.6 Elements of the Ideal Field

  • It represents a distinct characteristic of the subject of the table.
  • It contains only a single value.
  • It cannot be deconstructed into smaller components.
  • It does not contain a calculated or concatenated value.
  • It is unique within the entire database structure.
  • It retains a majority of its properties when it appears in more than one table.

1.3.5.7 Elements of the Ideal Table

  • It represents a single subject, which can be an object or event.
  • It has a primary key.
  • It does not contain multipart or multivalued fields.
  • It does not contain calculated fields.
  • It does not contain unnecessary duplicate fields. (does not apply to fields used to relate tables together)
  • It contains only an absolute minimum amount of redundant data.

1.3.7 Ch. 9

1.3.7.1 Field Specifications

Field Specifications are crucial because

  • Field specifications help establish and enforce field-level integrity.
  • Defining field specifications for each field enhances overall data integrity.
  • Defining field specifications compels you to acquire a complete understanding of the nature and purpose of the data in the database.
  • Field specifications constitute the “data dictionary” of the database.

a field reaches Field-level integrity if

  • The identity and purpose of a field are clear, and all the tables in which it appears are properly identified.
  • Field definitions are consistent throughout the database.
  • The values of a field are consistent and valid.
  • The types of modifications that can be applied to the values in the field are clearly identified.

1.3.7.2 Anatomy of a Field Specification

1.3.7.2.1 General Elements

Field Name, Parent Table, Specification Type, Source Specification, Shared By, Alias(es), Description

The most fundamental attributes of the field

The Field Name is the set of absolutely minimal words that uniquely identifies a particular fields throughtout the database

A table that incorporates a given field within its structure is known as the fields parent table this is the only table w/i which the field will appear unless the field is participating in establishing a relationship.

the elements set for a given field depend n the type of specification defined for the field. it can be defined in three ways

  1. Unique the default unless a template or foreign key, can receive all but Source Specification element and elements only apply to specified field from the Field Name element
  2. Generic template field to help ensure consistent definitions for fields with the same general meaning,(ex. for different state fields, even if they have different subjects they all contain a US State) requires a nonspecific field name and element settings that are as broad and general as possible but can take all elements but Parent Table, Shared By, Alias(es), and Source Specification
  3. Replica this is the default for a field based on a generic field or that serves as a foreign key, it draws the majority of its element settings from an existing specification but elements can be altered or added

the Source Specification element is set only on a Replica specification and indicates the name of the specific field specification upon which the current specification is based.

the Shared By element indicates the names of other tables that share this field, the only table names that showuld appear hear are those that have an explicit relationship to the fields parent table

the Alias(es) element is a name or set of names that you use for the field in very rare circumstances for instance where there must be two instances of the field in the same table

the description element is a complete interpretation of the field

1.3.7.2.2 Physical Elements

Data Type, Length, Decimal Places, Character Support

The physical elements category pertains to the structure of a field its elements are expressed in general terms because each RDBMS program implements them in a slightly different manner.

the Data Type elelment indicates the nature of data that the field stores there are 3 main types (more exist and are used but they tend to be more program specific)

  1. Alphanumeric - any combo including keyboard and special characters
  2. Numeric - only whole and real numbers, not including numbers with leading 0s (ex. 0022)
  3. DateTime - dates times or combo of both

the Length element specifies the total number of characters that a user can enter for any given field value (some programs do not allow setting of this value)

the Decimal places element denotes the number of digits to the right of the decimal point in a real number.

the character support element indicates the types if characters that a user can enter into a given field value - can include or exclude any of the following types of characters

  • Letters: All letters of the alphabet including foreign language letters such as é and ñ.
  • Numbers: 0 through 9.
  • Keyboard characters: Any standard character other than letters and numbers, such as asterisk, ampersand, bracket, caret, comma, equals sign, exclamation point, parenthesis, percent sign, period, pound sign, question mark, quote, semicolon, slash, or vertical bar.
  • Special characters: Any character that you can produce only through specific combinations of standard keys and the CTRL, ALT, and/or SHIFT keys, or with the aid of a special software program. Characters in this category include complex mathematical symbols, the copyright symbol, fractions, the symbol for pi, and the trademark symbol.
1.3.7.2.3 Logical Elements

Key Type, Key Structure, Uniqueness, Null Support, Values Entered By, Required Value, Range of Values, Edit Rule

the Logical Elements categor pertains mainly to the values within a field, governing matters such as role w/i table, uniqueness, when in should be entered, can it be edited…

the Key type element designates a fields role w/i a table (non-key, primary key, alternate key, foreign key)

the Key structure element denotes whether a field designates as a primary key is acting as a simple (single-field) or part of a composite (multifield) primary key

the Uniqueness element indicates whether a fields values are unique (typically just for primary key) or non-unique

the Null support element specifies whether a field excepts nulls or not

the Values entered by element indicates the source of the fields values (needs to be set up before works)

the Required value element denotes whether a user is required to enter a value for a field (must be yes for the primary key)

the Range of values element specifies every possible valid value for this field, this range can be established under three general categories. (DO NOT use “other” or “miscellaneous” w/i ANY category)

  1. General - A complete collection of every possible value for this field.
  2. Integrity-specific - A collection of values based on the field’s role within a table relationship.
  3. Business-specific - A collection of values generated by a particular business requirement.

the Edit rule element designates at what point a user can enter a value into a field and whether they can modify that value has 5 options

  1. Enter Now, Edits Allowed - must enter value when creating a new record, value can be edited at any time
  2. Enter Later, Edits Allowed - can enter when creating a new record, must at some point, value can then be edited at any time
  3. Enter Now, Edits Not Allowed - must enter value when creating new record, cannot be edited
  4. Enter Later, Edits Not Allowed - can enter when creating a new record, must at some point, value cannot be edited
  5. Not Determined At This Time - the edit rule will be determined when the database is implemented in an RDBMS program

1.3.7.3 Guidelines for Composing a Field Description

  • Use a statement that accurately identifies the field and clearly states its purpose.
  • Write a clear and succinct statement.
  • Refrain from restating or rephrasing the field name.
  • Avoid using technical jargon, acronyms, or abbreviations.
  • Do not include implementation-specific information.
  • Do not make this description dependent upon the description of another field.
  • Do not use examples.

1.3.8.2 Resolving a Multivalued Field

  1. Remove the field from the table and use it as the basis for a new table. If necessary, rename the field in accordance with the field naming guidelines that you learned earlier in this chapter.

1.3.9 Ch. 11

A business rule is a statement that imposes some form of constraint on a specific aspect of the database, such as the elements within a field specification for a particular field or the characteristics of a given relationship ex.

a ship date cannot be before an order date while this is technically possible within the DB adding this constraint allows the ship date to be an actually meaningful date vs. nonsense

You base a business rule on the way the organization perceives and uses its data, which you determine from the manner in which the organization functions or conducts its business.

Business rules often apply to the Degree of Participation between tables, which is often not able to be established as an in-database logical constraint

1.3.9.1 Types of business rules

  • The two major types of business rules are Database Oriented and Application Oriented. Both enforce constraints and data integrity, but differ in where and how they are established.
1.3.9.1.1 Database Oriented Business Rules

impose constraints you can establish w/i the logical design of the database by modifying field spec. elements and/or relationship char.

MUST be able to meaningfully and clearly establish constraint by either means

ex. noodles only exist in midwest allow only midwest states in range of values

1.3.9.1.2 Application Oriented Business Rules

impose constraints that you cannot establish w/i logical design of DB must instead be established w/i physical design or the database software application design

ex. members earn 15% discount, This business rule determines the amount of discount applied to a customer’s purchases, based on a particular status. You cannot establish this constraint meaningfully in the logical design for two reasons: There is no field in which to store the discount amount (the amount is a result of a calculation, and calculated fields are not allowed in a table), and there is no way to indicate the criterion used-the customer’s status-to determine the discount. You must establish this rule within the physical design of the database or the design of the database application.

1.3.9.2 Categories of business rules

1.3.9.2.1 Field-Specific Business rule

impose constraints on the elements of a field specification for a particular field. The number of elements a given rule affects depends on the manner in which you define that rule.

1.3.9.2.2 Relationship-Specific Business Rules

impose constraints that affect the characteristics of a relationship.

1.3.9.3 Defining and Establishing Field-Specific Business rules

  1. Select a table.
  2. Review each field and determine whether it requires any constraints.
  3. Define the necessary business rules for the field.
  4. Establish the rules by modifying the appropriate field specification elements.
  5. Determine what actions test the rule.
  6. Record the rule on a Business Rule Specifications sheet

1.3.9.4 Defining and Establishing relationship-Specific Business rules

  1. Select a relationship.
  2. Review the relationship and determine whether it requires any constraints.
  3. Define the necessary business rules for the relationship.
  4. Establish the rule by modifying the appropriate relationship characteristics.
  5. Determine what actions will test the rule.
  6. Record the rule on a Business Rule Specifications sheet.

1.4.10 Lesson 10 grouping

GROUP BY groups the data in the response by the specified column(s)

SELECT column_name(s)
FROM table_name
WHERE condition
GROUP BY column_name(s)
ORDER BY column_name(s);

You also want to use ORDER BY when grouping to get a coherent sorting

  • GROUP BY() clauses can contain as many columns as you want. This enables you to nest groups, providing you with more granular control over how data is grouped.
  • If you have nested groups in your GROUP BY clause, data is summarized at the last specified group. In other words, all the columns specified are evaluated together when grouping is established (so you won’t get data back for each individual column level).
  • Every column listed in GROUP BY must be a retrieved column or a valid expression (but not an aggregate function). If an expression is used in the SELECT, that same expression must be specified in GROUP BY. Aliases cannot be used.
  • Most SQL implementations do not allow GROUP BY columns with variable length datatypes (such as text or memo fields).
  • Aside from the aggregate calculation statements, every column in your SELECT statement must be present in the GROUP BY clause.
  • If the grouping column contains a row with a NULL value, NULL will be returned as a group. If there are multiple rows with NULL values, they’ll all be grouped together.
  • The GROUP BY clause must come after any WHERE clause and before any ORDER BY clause. although WHERE only looks at specific rows having no concept of groups

HAVING is what is needed in order to filter the grouping itself and all options for WHERE can be used for HAVING ex.

HAVING COUNT(*) > 2;
/* which returns the number of rows in
column1 that have more than 2 rows 
WHERE wouldn't work in this situation
because you want the filtering w/i the
grouping not before or after*/

\[ Rules \]

  • UNION must be composed of two or more SELECT statements, each separated by the keyword UNION and the columns MUST be in the same order

1.4.19.1 Primary Key Setting

to set the primary key when initially creating a table you include after your list of columns a primary key constraint, which can be one or multiple columns

#to set the primary key when one column  CREATE TABLE table_name( 
col_1 int NOT NULL, 
col_2 int, 
col_3 int, 
PRIMARY KEY (col1) );
#OR
CREATE TABLE table_name( 
col_1 int NOT NULL PRIMARY KEY, 
col_2 int, 
col_3 int);
#to set the primary key when multiple columns 
CREATE TABLE table_name( 
col_1 int NOT NULL, 
col_2 int NOT NULL, 
col_3 int, 
CONSTRAINT a_name PRIMARY KEY (col_1,col_2) ); 
/*this sets col_1 & 2 together as the VALUE of the primary key, but under one name*/

to set the primary key when altering a table, it just runs as a regular alter table command with the synax as above AND the column(s) being set as primary key MUST have been marked NOT NULL during creation ex.

ALTER TABLE table_name 
ADD PRIMARY KEY (col_name); 
#or the multiple column syntax above
#to drop a primary key constraint 
ALTER TABLE table_name 
DROP PRIMARY KEY;

1.4.19.3 Unique Constraints

unique constraints are used to ensure that all data in column(s) is unique. similar to PK (ex. you would want a unique constraint on a social security number but do not want to use that as the PK)

  • A table can contain multiple unique constraints, but only one primary key is allowed per table.
  • Unique constraint columns can contain NULL values.
  • Unique constraint columns can be modified or updated.
  • Unique constraint column values can be reused.
  • Unlike primary keys, unique constraints cannot be used to define foreign keys.

uses either UNIQUE in table definition or a separate CONSTRAINT
